# I313 — Android scan-response control UUID Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Route the Bluey lifecycle control UUID into Android's scan-response slot (a separate 31-byte buffer) instead of the primary advertisement, so `peerDiscoverable: true` no longer competes with user service UUIDs for the 31-byte primary-AD budget. iOS scanners filtering on the control UUID see scan-response UUIDs via active scan (the foreground default), so cross-platform peer discovery (Android-server → iOS-central) starts working.

**Architecture:** Add a `scanResponseServiceUuids` field through the existing layering — platform-interface DTO → per-platform Pigeon DTOs → native plugins. The domain layer (`BlueyServer.startAdvertising`) splits UUIDs: the user's services stay in the primary `serviceUuids` field, the control UUID (added when `peerDiscoverable: true`) goes to `scanResponseServiceUuids`. Android consumes the new field by feeding it into `AdvertiseData.Builder` for the second argument of `BluetoothLeAdvertiser.startAdvertising(settings, primary, scanResponse, callback)`. iOS accepts the field for DTO symmetry but ignores it — CoreBluetooth's overflow area already provides the equivalent cross-platform-visible secondary slot. While we're in the error path, surface `ADVERTISE_FAILED_DATA_TOO_LARGE` as the existing typed `AdvertisingException(AdvertisingFailureReason.dataTooBig)` instead of generic `bluey-unknown` so future regressions are obvious.

**Tech Stack:** Dart 3 (`bluey`, `bluey_platform_interface`, `bluey_android`, `bluey_ios`), Kotlin (Android plugin), Swift (iOS plugin), Pigeon for type-safe platform channels, `flutter_test` for Dart tests.

class PlatformAdvertiseConfig {
  final String? name;
  final List<String> serviceUuids;

  /// Service UUIDs to put in the scan-response packet (Android only).
  ///
  /// Android exposes a separate 31-byte scan-response buffer transmitted
  /// in response to active scans. Routing the Bluey lifecycle control
  /// UUID here keeps it out of the primary 31-byte budget so the user's
  /// app UUIDs aren't crowded out (see I313). Active scanners — including
  /// iOS CoreBluetooth's default centrals and Android's own scanner —
  /// receive scan-response data alongside the primary advertisement.
  ///
  /// Ignored on iOS: CoreBluetooth doesn't expose scan-response separately
  /// and its overflow area already provides the equivalent secondary slot.
  /// Defaults to an empty list.
  final List<String> scanResponseServiceUuids;

  final int? manufacturerDataCompanyId;
  final Uint8List? manufacturerData;
  final int? timeoutMs;

  /// The advertising mode (Android only).
  ///
  /// Controls the advertising interval and power consumption.
  /// Ignored on platforms other than Android.
  ///
  /// Defaults to [PlatformAdvertiseMode.lowLatency] if not specified.
  final PlatformAdvertiseMode? mode;

  const PlatformAdvertiseConfig({
    this.name,
    required this.serviceUuids,
    this.scanResponseServiceUuids = const [],
    this.manufacturerDataCompanyId,
    this.manufacturerData,
    this.timeoutMs,
    this.mode,
  });
}
```

/// Raised when the platform's advertising stack rejects an advertisement
/// because it exceeds the legacy 31-byte primary-AD budget (Android's
/// `ADVERTISE_FAILED_DATA_TOO_LARGE`, error code 1).
///
/// The domain layer translates this to `AdvertisingException(
/// AdvertisingFailureReason.dataTooBig)`. Surface this typed form
/// instead of generic `bluey-unknown` so apps can react (e.g., shorten
/// the device name, drop a UUID, or move it to scan response).
class PlatformAdvertiseDataTooLargeException implements Exception {
  final String message;

  const PlatformAdvertiseDataTooLargeException(this.message);

  @override
  String toString() =>
      'PlatformAdvertiseDataTooLargeException: $message';

  @override
  bool operator ==(Object other) =>
      identical(this, other) ||
      other is PlatformAdvertiseDataTooLargeException &&
          other.message == message;

  @override
  int get hashCode => message.hashCode;
}
```

## Task 9: iOS Pigeon DTO — accept `scanResponseServiceUuids` and fold into the unified advertisement

**Why iOS must USE the field, not ignore it.** CoreBluetooth doesn't
expose a separate scan-response API — there is one
`CBAdvertisementDataServiceUUIDsKey` that holds all advertised UUIDs;
iOS routes some to overflow when the primary 31-byte budget is
exceeded. Today the domain layer prepends the control UUID into
`serviceUuids` and iOS-to-Android cross-platform discovery works
because iOS puts the first-listed UUID in the primary slot.

After Task 3 the domain layer no longer prepends — it routes the
control UUID into `scanResponseServiceUuids` instead. If the iOS
plugin ignores that field, only the user's UUIDs reach
`CBAdvertisementDataServiceUUIDsKey`, the control UUID disappears from
the iOS advertisement entirely, and Android scanners filtering on the
control UUID stop seeing iOS — a regression of the currently-working
direction.

The fix: iOS folds `scanResponseServiceUuids` into the unified UUID
list, prepending so the control UUID retains primary-slot priority via
iOS overflow ordering. Wire-level intent is preserved across both
platforms; Android puts the control UUID in scan response, iOS puts it
in primary AD with overflow handling the rest.
