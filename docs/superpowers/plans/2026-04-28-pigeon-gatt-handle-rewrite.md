**Goal:** Implement the bundled rewrite of I088 (Pigeon GATT handle identity), I089/I066 (Android-tagged Connection extensions), I300 (PeerConnection composition), and I301 (value objects for Mtu / ConnectionParameters / handles). Major-version-bump breaking change; only the example app consumes Bluey, so no migration guide.

**Architecture:** Five phases ‚Äî value objects ‚Üí platform-tagged extensions ‚Üí peer composition ‚Üí Pigeon handle rewrite ‚Üí cleanup. Each phase ends at a review checkpoint where the suite is green, the example app builds, and the user manually verifies. Phase D (handle rewrite) is sequenced last so the handle work doesn't have to keep changing as Connection's surface settles.

**Tech Stack:** Dart 3, Flutter, Pigeon, Kotlin (Android), Swift (iOS), JVM unit tests (Robolectric / mockk), XCTest, FakeBlueyPlatform.

## Risk notes for the executor

- **Phase D.8 / D.13 ordering trap.** During the additive interim (D.8 through D.12), every Pigeon GATT method takes BOTH UUID and handle. Native receivers prefer handle when non-null. If you accidentally land D.13 (drop UUID) before all native sides are emitting handles correctly, builds break across both platforms simultaneously. Run JVM + XCTest after D.4, D.5, and D.6 to confirm handle emission before crossing into D.13.
- **Service Changed is rare in tests.** D.11 is a load-bearing correctness test but most fakes don't model Service Changed by default. The new test in D.11 may be the first; verify it actually exercises the post-discovery handle map rebuild rather than a no-op.
- **Capabilities matrix asymmetry.** B.2 hinges on capabilities being correct for the test platform. If a test sets up FakeBlueyPlatform with default capabilities (which may default to "Android-flavored" or vice versa), `connection.android` will be unexpectedly non-null/null. Always explicitly set capabilities in extension-related tests.
- **iOS handle counter is per-device.** `nextHandle: [String: Int]` is keyed by deviceId; if a device reconnects, the counter resets to 1 (because the dict entry was cleared on disconnect). That's the correct behavior ‚Äî the new connection has fresh handles. Don't accidentally make the counter global.
- **Android `getInstanceId()` is implementation-defined.** It returns an Int but the BLE Core spec doesn't mandate uniqueness across services in a peripheral's GATT database (though Android's stack does in practice). The handle table uses it as a key per-device; if a peripheral exposes two services that hash to the same instance ID, the handle table loses one. Our fallback: throw `AttributeNotFoundException` with a message that names the instance ID ‚Äî this is a "bug in your peripheral or in Android's stack" condition, not a Bluey bug, but be ready to spot it during D.checkpoint.5.
- **Don't `--no-verify` or `--no-edit`.** Per repo conventions, never skip hooks. If a pre-commit hook fails, fix the underlying issue; don't bypass.
- **Don't push.** User handles git pushes themselves (per `~/.claude/.../memory/feedback_no_push.md`).
